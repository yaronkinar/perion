Frontend Flow (Runtime)


1) Router enters page (e.g. login/dashboard)
2) Page composable runs (useLoginPage/useDashboardPage)
3) Service calls /api/* through Vite proxy
4) Pinia/session state updates from API envelope.data
5) Components re-render with derived permission state
6) Guarded UI paths are hidden/disabled declaratively

Data flow: Router -> Composable -> Service -> Store -> View.

Frontend Permission + Form Flow


Permission checks:
- usePermission().can(action)
- PermissionGuard(action, mode)
- Router meta.requireAny for route-level blocking

Form lifecycle:
- VeeValidate + Zod schema validates client-side
- submit -> backend DTO/class-validator checks
- errors[] mapped back to field-level UI state

Permission Model


Permissions:
- view_users
- create_user
- edit_user
- delete_user
- view_roles
- edit_roles
- change_role

Roles:
- Admin  -> all
- Editor -> view_users, edit_user, view_roles, change_role
- Viewer -> view_users

Backend Request Flow (NestJS)


HTTP Request
  -> PermissionsMiddleware (method+path -> required permission)
  -> Controller route handler
  -> Service business logic
  -> Repository/TypeORM + PostgreSQL
  -> TransformInterceptor wraps success response
  -> HttpExceptionFilter wraps and sanitizes failures
  -> HTTP Response { data, message, statusCode, errors? }

Backend Auth + Authorization Path


Login path:
POST /api/auth/login
-> verify email/password
-> create JWT (bonus path)
-> seed req.session.user snapshot (role + permissions)
-> set cookies and return authenticated user

Protected path:
Request enters protected users/roles route
-> PermissionsMiddleware resolves required permission by method+path
-> reads req.session.user.permissions
-> allow or deny

Sequence: Login (Email + Password)


Browser/LoginPage      Frontend API Service      Nest AuthController       AuthService/DB
      |                        |                        |                         |
1) user submits credentials    |                        |                         |
      |------ POST /api/auth/login -------------------> |                         |
      |                        |                        |-- verify user + hash -->|
      |                        |                        |<----- user row ----------|
      |                        |                        |-- create JWT + session -->|
      |<----- {data:{token,user}} + cookies -----------|                         |
2) store session user + permissions snapshot
3) router pushes to dashboard; guarded UI now enabled

Sequence: Users List Page Load


Browser/Dashboard   useDashboardPage   users.service   PermissionsMiddleware   UsersController   UsersService/DB
      |                    |                |               |                 |                |
1) route enter            |                |               |                 |                |
      |------------------->|               |               |                 |                |
2) fetchUsers()           |               |               |                 |                |
      |-------------------- GET /api/users ------------------------------->|                 |
      |                    |                |-- can(view_users)? -------->|                 |
      |                    |                |<-------- allow/deny --------|                 |
      |                    |                |----------------------------->|                 |
      |                    |                |               |------ query users ------------->|
      |                    |                |               |<---------- rows ---------------|
      |<-------- envelope { data: users[] }|               |                 |                |
3) table renders; Viewer role gets user list without role field

Sequence: Update Role Permissions


Role Editor UI       roles.service      PermissionsMiddleware   RolesController   RolesService/DB
      |                   |                  |                    |                      |
1) select role + permissions                |                    |                      |
2) submit PUT /api/roles/:id -------------->|                    |                      |
      |                   |-- can(edit_roles)? ------------------>|                      |
      |                   |<------------------- allow/deny -------|                      |
      |                   |-------------------------------------->|                      |
      |                   |                  |------ replace permission set ------------->|
      |                   |                  |<----------- updated role ------------------|
      |<-------- envelope { data: role } ---|                    |                      |
3) frontend refreshes roles state and reflects new matrix
4) existing logged-in sessions keep old snapshot until refresh/login

Sequence: Error Paths (403 + 400)


A) Permission denied (403)
Browser -> PUT /api/roles/:id
PermissionsMiddleware checks can(edit_roles)?
PermissionsMiddleware denies -> HttpExceptionFilter wraps:
{ data: null, message: "Forbidden", statusCode: 403 }
Frontend handles response and keeps restricted UI state

B) Validation failure (400)
Browser -> POST /api/users (invalid payload)
DTO/class-validator throws validation errors[]
HttpExceptionFilter returns envelope with errors[]
Frontend maps errors[] -> setFieldError(field, message)
User sees inline field-level messages in modal form

Frontend Guard Pattern


<PermissionGuard action="create_user" mode="disable">
  <button class="btn-primary">Add User</button>
</PermissionGuard>

UI behavior is declarative: hide or disable actions by permission.

Backend Auth + Session Snapshot


// Login seeds req.session.user with role + permission snapshot
POST /api/auth/login
-> returns { token, user }
-> sets session cookie (+ perion.jwt cookie)

PermissionsMiddleware reads permissions from session
for route authorization without per-request DB lookup.

Core API Surface


GET    /api/users         (view_users)
POST   /api/users         (create_user)
PUT    /api/users/:id     (edit_user)
DELETE /api/users/:id     (delete_user)
GET    /api/roles         (view_roles)
PUT    /api/roles/:id     (edit_roles)
GET    /api/auth/me       (session)
POST   /api/auth/select   (dev-only)
POST   /api/auth/login    (email/password + JWT)

Testing Strategy

  • Backend unit tests (Jest)
  • Frontend unit tests (Vitest)
  • E2E flows (Playwright)
  • CI runs all suites via Docker Compose

docker compose -f docker-compose.test.yml --profile unit up --build
docker compose -f docker-compose.test.yml --profile e2e up --build

Key Engineering Decisions

  • Default-deny permission middleware on protected routes
  • Consistent API envelope for success/errors
  • Session snapshot model for fast authorization checks
  • Docker-first local dev and CI parity
